# 来自用户反馈的 Skills 改进

**日期:** 2025-11-28
**状态:** 草稿
**来源:** 两个 Claude 实例在实际开发场景中使用 superpowers

---

## 执行摘要

两个 Claude 实例来自实际开发会话的详细反馈。他们的反馈揭示了当前 skills 中的**系统性缺陷**，这些缺陷导致可预防的错误在遵循 skills 的情况下仍然被发布。

**关键洞察:** 这些是问题报告，而不仅仅是解决方案提案。问题是真实的；解决方案需要仔细评估。

**关键主题:**
1. **验证缺陷** - 我们验证操作成功但不验证它们达到预期结果
2. **流程卫生** - 后台进程累积并在 subagents 之间产生干扰
3. **上下文优化** - Subagents 获得太多不相关信息
4. **缺乏自我反思** - 交接前没有提示批评自己的工作
5. **Mock 安全性** - Mocks 可以在无检测的情况下与接口偏离
6. **Skill 激活** - Skills 存在但没有被读取/使用

---

## 已识别的问题

### 问题 1: 配置更改验证缺陷

**发生了什么:**
- Subagent 测试 "OpenAI 集成"
- 设置 `OPENAI_API_KEY` 环境变量
- 获得状态 200 响应
- 报告 "OpenAI 集成工作"
- **但** 响应包含 `"model": "claude-sonnet-4-20250514"` - 实际上使用的是 Anthropic

**根本原因:**
`verification-before-completion` 检查操作成功但不检查结果反映预期的配置更改。

**影响:** 高 - 对集成测试的错误信心，错误发布到生产环境

**失败模式示例:**
- 切换 LLM 提供商 → 验证状态 200 但不检查模型名称
- 启用功能标志 → 验证无错误但不检查功能处于活动状态
- 更改环境 → 验证部署成功但不检查环境变量

---

### 问题 2: 后台进程累积

**发生了什么:**
- 会话期间 dispatch 多个 subagents
- 每个都启动后台服务器进程
- 进程累积（4+ 个服务器运行）
- 陈旧进程仍绑定到端口
- 后来的 E2E 测试使用错误配置命中陈旧服务器
- 令人困惑/不正确的测试结果

**根本原因:**
Subagents 是无状态的 - 不知道先前 subagents 的进程。没有清理协议。

**影响:** 中-高 - 测试使用错误的服务器，错误的通过/失败，调试混乱

---

### 问题 3: Subagent 提示中的上下文膨胀

**发生了什么:**
- 标准方法: 给 subagent 完整的计划文件来读取
- 实验: 仅给任务 + 模式 + 文件 + 验证命令
- 结果: 更快、更专注，单次完成更常见

**根本原因:**
Subagents 在无关的计划部分上浪费 token 和注意力。

**影响:** 中 - 执行较慢，更多失败的尝试

**有效的方法:**
```
你正在向 packnplay 的测试套件添加单个 E2E 测试。

**你的任务:** 将 `TestE2E_FeaturePrivilegedMode` 添加到 `pkg/runner/e2e_test.go`

**要测试什么:** 在其元数据中请求 `"privileged": true` 的本地 devcontainer 功能应导致容器使用 `--privileged` 标志运行。

遵循 TestE2E_FeatureOptionValidation 的确切模式（在文件末尾）

**写入后，运行:** `go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m`
```

### 问题 4: 交接前无自我反思

**发生了什么:**
- 添加了自我反思提示: "用新鲜的眼睛审视你的工作 - 什么可以更好？"
- 任务 5 的实施者识别出失败的测试是由于实施错误，而不是测试错误
- 追踪到第 99 行: `strings.Join(metadata.Entrypoint, " ")` 创建了无效的 Docker 语法
- 如果没有自我反思，只会报告 "测试失败" 而没有根本原因

**根本原因:**
实施者在报告完成之前不会自然地退后一步并批评自己的工作。

**影响:** 中 - 错误交给审查者，实施者本可以捕获的错误

### 问题 5: Mock-接口漂移

**发生了什么:**
```typescript
// 接口定义 close()
interface PlatformAdapter {
  close(): Promise<void>;
}

// 代码（错误）调用 cleanup()
await adapter.cleanup();

// Mock（匹配错误）定义 cleanup()
vi.mock('web-adapter', () => ({
  WebAdapter: vi.fn().mockImplementation(() => ({
    cleanup: vi.fn().mockResolvedValue(undefined),  // 错误！
  })),
}));
```
- 测试通过
- 运行时崩溃: "adapter.cleanup is not a function"

**根本原因:**
Mock 从错误代码调用的内容派生，而不是从接口定义派生。TypeScript 无法使用错误的方法名称捕获内联 mocks。

**影响:** 高 - 测试给出错误的信心，运行时崩溃

**为什么 testing-anti-patterns 没有防止这种情况:**
skill 涵盖了测试 mock 行为和在无理解的情况下 mocking，但没有涵盖 "从接口派生 mock，而不是实现" 的特定模式。

### 问题 6: 代码审查者文件访问

**发生了什么:**
- dispatch 了代码审查者 subagent
- 找不到测试文件: "文件似乎在仓库中不存在"
- 文件实际存在
- 审查者不知道需要先显式读取它

**根本原因:**
审查者提示不包含显式文件读取说明。

**影响:** 低-中 - 审查失败或不完整

---

### 问题 7: 修复工作流延迟

**发生了什么:**
- 实施者在自我反思期间识别错误
- 实施者知道修复方法
- 当前工作流: 报告 → 我 dispatch 修复者 → 修复者修复 → 我验证
- 额外的往返增加了延迟而没有增加价值

**根本原因:**
当实施者已经诊断时，实施者和修复者角色之间的刚性分离。

**影响:** 低 - 延迟，但没有正确性问题

---

### 问题 8: Skills 未被读取

**发生了什么:**
- `testing-anti-patterns` skill 存在
- 无论是人类还是 subagents 在编写测试之前都没有读取它
- 本可以防止一些问题（尽管不是全部 - 见问题 5）

**根本原因:**
没有强制执行 subagents 读取相关的 skills。没有提示包括 skill 读取。

**影响:** 中 - 如果不使用，skill 投资将被浪费

## 拟议的改进

### 1. verification-before-completion: 添加配置更改验证

**添加新部分:**

```markdown
## 验证配置更改

当测试配置、提供商、功能标志或环境的更改时:

**不要只验证操作成功。验证输出反映预期的更改。**

### 常见失败模式

操作成功是因为*某些*有效配置存在，但不是你打算测试的配置。

### 示例

| 更改 | 不充分 | 必需 |
|--------|-------------|----------|
| 切换 LLM 提供商 | 状态 200 | 响应包含预期模型名称 |
| 启用功能标志 | 无错误 | 功能行为实际处于活动状态 |
| 更改环境 | 部署成功 | 日志/变量引用新环境 |
| 设置凭据 | 身份验证成功 | 已验证的用户/上下文正确 |

### 门控函数

```
在声称配置更改工作之前:

1. 识别: 此更改后应该有什么不同？
2. 定位: 该差异在哪里可观察？
   - 响应字段（模型名称、用户 ID）
   - 日志行（环境、提供商）
   - 行为（功能活动/不活动）
3. 运行: 显示可观察差异的命令
4. 验证: 输出包含预期差异
5. 只有这样: 声称配置更改工作

红旗:
  - "请求成功" 而不检查内容
  - 检查状态代码但不检查响应体
  - 验证无错误但不进行积极确认
```

**为什么有效:**
强制验证意图，而不仅仅是操作成功。

### 2. subagent-driven-development: 为 E2E 测试添加流程卫生

**添加新部分:**

```markdown
## E2E 测试的流程卫生

当 dispatch 启动服务（服务器、数据库、消息队列）的 subagents 时:

### 问题

Subagents 是无状态的 - 它们不知道先前 subagents 启动的进程。后台进程持续存在并可能干扰后续测试。

### 解决方案

**在 dispatch E2E 测试 subagent 之前，在提示中包含清理:**

```
在启动任何服务之前:
1. 杀死现有进程: pkill -f "<service-pattern>" 2>/dev/null || true
2. 等待清理: sleep 1
3. 验证端口空闲: lsof -i :<port> && echo "ERROR: Port still in use" || echo "Port free"

测试完成后:
1. 杀死你启动的进程
2. 验证清理: pgrep -f "<service-pattern>" || echo "Cleanup verified"
```

### 示例

```
任务: 运行 API 服务器的 E2E 测试

提示包括:
"在启动服务器之前:
- 杀死任何现有服务器: pkill -f 'node.*server.js' 2>/dev/null || true
- 验证端口 3001 空闲: lsof -i :3001 && exit 1 || echo 'Port available'

测试后:
- 杀死你启动的服务器
- 验证: pgrep -f 'node.*server.js' || echo 'Cleanup verified'"
```

### 为什么这很重要

- 陈旧进程使用错误配置提供请求
- 端口冲突导致静默失败
- 进程累积减慢系统
- 令人困惑的测试结果（使用错误的服务器）
```

**权衡分析:**
- 向提示添加样板代码
- 但防止非常令人困惑的调试
- 对于 E2E 测试 subagents 值得

### 3. subagent-driven-development: 添加精简上下文选项

**修改步骤 2: 使用 Subagent 执行任务**

**之前:**
```
从 [plan-file] 仔细读取该任务。
```

**之后:**
```
## 上下文方法

**完整计划（默认）:**
当任务复杂或有依赖关系时使用:
```
从 [plan-file] 仔细读取任务 N。
```

**精简上下文（用于独立任务）:**
当任务是独立的且基于模式时使用:
```
你正在实现: [1-2 句话任务描述]

要修改的文件: [精确路径]
要遵循的模式: [对现有函数/测试的引用]
要实现的内容: [具体要求]
验证: [要运行的精确命令]

[不要包含完整计划文件]
```

**在以下情况使用精简上下文:**
- 任务遵循现有模式（添加类似测试、实现类似功能）
- 任务是自包含的（不需要其他任务的上下文）
- 模式引用足够（例如，"遵循 TestE2E_FeatureOptionValidation"）

**在以下情况使用完整计划:**
- 任务依赖于其他任务
- 需要理解整体架构
- 需要上下文的复杂逻辑
```

**示例:**
```
精简上下文提示:

"你正在为 devcontainer 功能添加特权模式测试。

文件: pkg/runner/e2e_test.go
模式: 遵循 TestE2E_FeatureOptionValidation（在文件末尾）
测试: 元数据中带有 `"privileged": true` 的功能导致 `--privileged` 标志
验证: go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m

报告: 实施、测试结果、任何问题。"
```

**为什么有效:**
在适当时减少 token 使用，增加专注，更快完成。

### 4. subagent-driven-development: 添加自我反思步骤

**修改步骤 2: 使用 Subagent 执行任务**

**添加到提示模板:**

```
完成后，在报告之前:

退后一步，用新鲜的眼睛审视你的工作。

问自己:
- 这是否真正解决了指定的任务？
- 有我没有考虑的边缘情况吗？
- 我是否正确遵循了模式？
- 如果测试失败，根本原因是什么（实施错误 vs 测试错误）？
- 这个实施有什么可以改进的？

如果你在反思期间识别到问题，现在修复它们。

然后报告:
- 你实施的内容
- 自我反思发现（如果有）
- 测试结果
- 更改的文件
```

**为什么有效:**
在交接之前捕获实施者可以自己发现的错误。有记录的案例: 通过自我反思识别入口点错误。

**权衡:**
每个任务增加约 30 秒，但在审查之前捕获问题。

### 6. testing-anti-patterns: 添加 Mock-接口漂移反模式

**添加新的反模式 6:**

```markdown
## 反模式 6: 从实现派生的 Mocks

**违规行为:**
```typescript
// 代码（错误）调用 cleanup()
await adapter.cleanup();

// Mock（匹配错误）有 cleanup()
const mock = {
  cleanup: vi.fn().mockResolvedValue(undefined)
};

// 接口（正确）定义 close()
interface PlatformAdapter {
  close(): Promise<void>;
}
```

**为什么这是错误的:**
- Mock 将 bug 编码到测试中
- TypeScript 无法使用错误的方法名称捕获内联 mocks
- 测试通过是因为代码和 mock 都是错误的
- 运行时使用真实对象时崩溃

**修复:**
```typescript
// ✅ 好: 从接口派生 mock

// 步骤 1: 打开接口定义（PlatformAdapter）
// 步骤 2: 列出那里定义的方法（close, initialize 等）
// 步骤 3: 仅 mock 这些确切的方法

const mock = {
  initialize: vi.fn().mockResolvedValue(undefined),
  close: vi.fn().mockResolvedValue(undefined),  // 从接口！
};

// 现在测试失败，因为代码调用不存在于 mock 中的 cleanup()
// 该失败在运行时之前揭示了错误
```

### 门控函数

```
在编写任何 mock 之前:

  1. 停止 - 尚未查看被测试的代码
  2. 查找: 依赖的接口/类型定义
  3. 读取: 接口文件
  4. 列出: 接口中定义的方法
  5. Mock: 仅这些具有确切名称的方法
  6. 不要: 查看你的代码调用的内容

  如果你的测试因为代码调用不在 mock 中的内容而失败:
    ✅ 好 - 测试在你的代码中发现了错误
    修复代码以调用正确的接口方法
    而不是修改 mock

  红旗:
    - "我将 mock 代码调用的内容"
    - 从实现复制方法名称
    - 在不读取接口的情况下编写 mock
    - "测试失败所以我要将此方法添加到 mock"
```

**检测:**

当你在运行时看到错误 "X is not a function" 而测试通过时:
1. 检查 X 是否被 mock
2. 比较 mock 方法与接口方法
3. 查找方法名称不匹配
```

**为什么有效:**
直接解决反馈中的失败模式。

### 7. subagent-driven-development: 要求测试 Subagents 读取 Skills

**在任务涉及测试时添加到提示模板:**

```markdown
在编写任何测试之前:

1. 读取 testing-anti-patterns skill:
   使用 Skill 工具: superpowers:testing-anti-patterns

2. 在以下情况应用该 skill 中的门控函数:
   - 编写 mocks
   - 向生产类添加方法
   - Mocking 依赖

这不是可选的。违反反模式的测试将在审查中被拒绝。
```

**为什么有效:**
确保 skills 实际被使用，而不仅仅是存在。

**权衡:**
为每个任务增加时间，但防止整类错误。

### 8. subagent-driven-development: 允许实施者修复自我识别的问题

**修改步骤 2:**

**当前:**
```
Subagent 报回工作摘要。
```

**拟议:**
```
Subagent 执行自我反思，然后:

如果自我反思识别到可修复的问题:
  1. 修复问题
  2. 重新运行验证
  3. 报告: "初始实施 + 自我反思修复"

否则:
  报告: "实施完成"

在报告中包括:
- 自我反思发现
- 是否应用了修复
- 最终验证结果
```

**为什么有效:**
当实施者已经知道修复方法时减少延迟。有记录的案例: 将为入口点错误节省一次往返。

**权衡:**
稍微更复杂的提示，但端到端更快。

## 风险和缓解措施

### 风险: 提示膨胀
**问题:** 添加所有这些要求使提示变得难以承受
**缓解措施:**
- 分阶段实施（不要一次添加所有内容）
- 使某些添加有条件（E2E 卫生仅用于 E2E 测试）
- 考虑不同任务类型的模板

### 风险: 分析瘫痪
**问题:** 太多的反思/验证减慢执行
**缓解措施:**
- 保持门控函数快速（秒，而不是分钟）
- 最初使精简上下文可选
- 监控任务完成时间

### 风险: 虚假的安全感
**问题:** 遵循清单不能保证正确性
**缓解措施:**
- 强调门控函数是最小值，而不是最大值
- 在 skills 中保持 "使用判断" 语言
- 记录 skills 捕获常见失败，而不是所有失败

### 风险: Skill 分歧
**问题:** 不同的 skills 给出冲突的建议
**缓解措施:**
- 审查所有 skills 的更改以保持一致性
- 记录 skills 如何交互（集成部分）
- 在部署前使用真实场景进行测试
