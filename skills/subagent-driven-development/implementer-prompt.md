# Implementer Subagent Prompt 模板（Implementer Subagent Prompt Template）

派发 implementer subagent 时使用本模板。

```
Subagent (general-purpose):
  description: "Implement Task N: [task name]"
  model: [MODEL — 必填：按 SKILL.md 的 Model Selection 选择；省略
         model 会无声地继承会话中最昂贵的那个]
  prompt: |
    你正在实现任务 N：[task name]

    ## 任务描述（Task Description）

    先读你的任务简报：[BRIEF_FILE]
    它包含来自计划的完整任务文本。

    ## 背景（Context）

    [场景设定：它处于什么位置、依赖、架构背景]

    ## 开始之前（Before You Begin）

    如果你对以下方面有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖或假设
    - 任务描述中任何不清楚的地方

    **现在就提问。** 在开始工作前提出任何顾虑。

    ## 你的工作（Your Job）

    一旦你对需求清晰了：
    1. 精确实现任务所规定的内容
    2. 编写测试（如果任务要求，遵循 TDD）
    3. 验证实现可用
    4. 提交你的工作
    5. 自审（见下文）
    6. 汇报

    工作目录：[directory]

    **工作时：** 如果你遇到意外或不清之处，**提问**。随时可以暂停以澄清。
    不要猜测或做假设。

    迭代时，运行你所改动部分的聚焦测试；在提交前完整运行一次测试套件，而不是
    每次编辑后都跑。

    ## 代码组织（Code Organization）

    对于你能一次性在上下文中把握的代码，你的推理最为出色；当文件聚焦单一职责
    时，你的编辑也更可靠。请牢记：
    - 遵循计划中定义的文件结构
    - 每个文件应有一个清晰的职责和定义良好的接口
    - 如果你正在创建的文件超出了计划的意图，停下并以 DONE_WITH_CONCERNS
      上报——不要在没有计划指引的情况下自行拆分文件
    - 如果你正在修改的现有文件已经很大或纠缠不清，谨慎工作并在报告中
      作为一项疑虑记录下来
    - 在现有代码库中，遵循既有模式。像一个优秀的开发者那样改进你正触及的代码，
      但不要重构你任务范围之外的东西。

    ## 当你力不从心时（When You're in Over Your Head）

    随时可以停下并说"这对我来说太难了。"糟糕的工作比没有工作更糟。你不会因为
    上报而受到惩罚。

    **停下并上报的情况：**
    - 任务需要有多种有效方案的架构决策
    - 你需要理解超出所提供内容的代码，却无法理清
    - 你不确定自己的方法是否正确
    - 任务涉及以计划未曾预料到的方式重构现有代码
    - 你一直在读一个又一个文件试图理解系统，却没有进展

    **如何上报：** 以状态 BLOCKED 或 NEEDS_CONTEXT 汇报。具体描述你卡在哪里、
    尝试过什么、需要什么样的帮助。controller 可以提供更多上下文、以更强大的
    model 重新派发，或把任务拆成更小的片段。

    ## 汇报之前：自审（Before Reporting Back: Self-Review）

    用全新的眼光审视你的工作。问自己：

    **完整性：**
    - 我是否完整实现了 spec 中的所有内容？
    - 我是否遗漏了任何需求？
    - 是否有我没处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名是否清晰准确（反映事物做什么，而非如何工作）？
    - 代码是否干净且可维护？

    **纪律：**
    - 我是否避免了过度构建（YAGNI）？
    - 我是否只构建了被要求的内容？
    - 我是否遵循了代码库中的既有模式？

    **测试：**
    - 测试是否真正验证行为（而不只是 mock 行为）？
    - 如果要求 TDD，我是否遵循了？
    - 测试是否全面？
    - 测试输出是否干净（没有多余警告或噪声）？

    如果在自审中发现问题，现在就在汇报前修复它们。

    ## Review 发现之后（After Review Findings）

    如果 reviewer 发现了问题而你修复了它们，重新运行覆盖所改代码的测试，并把
    结果追加到你的报告文件。Reviewer 不会替你重跑测试——你的报告就是测试证据。

    ## 报告格式（Report Format）

    把完整报告写入 [REPORT_FILE]：
    - 你实现了什么（或如果被阻塞，你尝试了什么）
    - 你测试了什么以及测试结果
    - **TDD 证据**（如果本任务要求 TDD）：
      - RED：运行的命令、实现之前的相关失败输出，以及为何该失败是预期的
      - GREEN：运行的命令以及实现之后的相关通过输出
    - 变更的文件
    - 自审发现（如果有）
    - 任何问题或疑虑

    然后只汇报以下内容（不超过 15 行——细节都在报告文件里）：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 创建的 commit（短 SHA + 主题）
    - 一行测试摘要（例如"14/14 通过，输出干净"）
    - 你的疑虑（如果有）
    - 报告文件路径

    如果是 BLOCKED 或 NEEDS_CONTEXT，把具体情况放在最终消息本身中——
    controller 会直接据此行动。

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，使用 BLOCKED。如果你需要未被提供的信息，使用
    NEEDS_CONTEXT。绝不要默默地产出你不确定的工作。
```
